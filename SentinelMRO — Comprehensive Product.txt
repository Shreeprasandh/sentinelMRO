Technical Stack ArchitectureBackend & Machine Learning Framework: Python 3.11+, FastAPI, PyTorch (Training), ONNX Runtime (Quantized Edge Inference), NumPy, Pandas, Scikit-learn.Dataset Matrix: NASA C-MAPSS Turbofan Engine Degradation Dataset (FD001 engine trajectories; 21 sensor channels, 3 operational settings).Federated Optimization Infrastructure: Custom multi-threaded/process loop executing the FedProx optimization algorithm with a tunable proximal parameter ($\mu$).Security & Immutable Ledger: Ed25519 cryptographic signing keys for simulated mTLS node validation; an inline Merkle Mountain Range (MMR) append-only cryptographic log backed by SQLite.Frontend Interface: Next.js 15 (App Router), React 19, Tailwind CSS, Lucide Icons, Shadcn/ui components, Recharts for high-frequency time-series data visualization.Execution Directives: Phase-by-PhasePhase 1: Data Preprocessing & Quantized Temporal Convolutional Network (TCN)Dataset Preprocessing & Target Labeling:Ingest train_FD001.txt and test_FD001.txt. Clean the 21 raw sensor channels by stripping low-variance constants (Sensors 1, 5, 6, 10, 16, 18, 19).Calculate Piecewise Linear Remaining Useful Life (RUL) targets: set maximum RUL constraints to $125$ cycles to accurately reflect early-stage operating plateaus before exponential degradation.Normalize sensor outputs using a localized MinMaxScaler bound strictly within a $[0, 1]$ structural limit. Apply a rolling window slice size of $30$ past operational cycles.Model Architecture (TCN over LSTM):Build a Temporal Convolutional Network (TCN) using residual blocks containing dilated 1D convolutions, weight normalization, and causal gating layers. Dilations scale exponentially ($1, 2, 4, 8$) with a kernel size of $k=3$.Reasoning: TCNs offer significantly longer effective historical horizons and faster execution speeds than standard LSTMs because convolutions can be computed in parallel.Quantization Pipeline:Export the fully trained PyTorch TCN model to an ONNX intermediate graph. Use ONNX Runtime’s toolset to apply post-training INT8 quantization (quantize_dynamic).FastAPI Endpoint Deployment: Create an endpoint at /api/v1/edge/inference that accepts a $30 \times 14$ matrix payload. It returns the current continuous health score ($1.0 \to 0.0$), the predicted RUL integer, and a boolean anomaly flag triggered if the current reconstruction loss or RUL falls below a specific statistical threshold ($RUL < 30$).Phase 2: Heterogeneous Federated Learning Simulation (FedProx & LDP)Non-IID Data Partitioning:Divide the C-MAPSS dataset into 3 separate operational clusters (simulating different MRO stations).Station 1: Primarily nominal operations (early cycles). Station 2: High-stress operational profiles (rapidly degrading engines). Station 3: Mixed environments. This layout directly mimics real-world non-IID conditions.FedProx Core Loop Implementation:Instead of standard FedAvg, implement the FedProx mathematical optimization routine. During local optimization epochs, modify the loss function by adding an explicit proximal regularization penalty:$$\mathcal{L}_{\text{prox}}(w) = \mathcal{L}_{\text{local}}(w) + \frac{\mu}{2} \| w - w^t \|^2$$where $w^t$ represents the current weights of the global model and $\mu = 0.01$. This keeps local weight adjustments stable when dealing with diverse datasets across stations.Local Differential Privacy (LDP):Before any local weights are sent back to the aggregator, add calibrated Gaussian noise directly to the model gradients:$$w_{\text{priv}} = w_{\text{local}} + \mathcal{N}(0, \sigma^2)$$This technique provides mathematical insurance against model inversion attacks, ensuring that no raw telemetry data can be reconstructed from the gradients.Tracking & Metrics Endpoint:Expose an endpoint at /api/v1/federated/aggregate that updates the global model state and returns a comprehensive metrics payload containing the current federation round, participant station keys, local loss metrics, and the current global validation loss.Phase 3: Cryptographic Merkle Mountain Range (MMR) Audit LedgerCryptographic Handshake Simulation:Every payload sent from an edge station to the aggregator must include a signature in the header, generated using the station’s private Ed25519 key. The API gateway validates this signature against the station's known public key to securely mock mTLS validation.Structure of the Ledger:Build a Merkle Mountain Range (MMR) ledger inside a local SQLite database file (ledger.db).Reasoning: Unlike a standard linear hash chain that requires $O(n)$ verification lookups, an MMR structures nodes into an append-only binary tree format. This enables incredibly fast $O(\log n)$ cryptographic validation and supports efficient proofs of inclusion.Data Schema Definitions:SQLCREATE TABLE mro_ledger (
